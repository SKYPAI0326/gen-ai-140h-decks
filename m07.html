The Three · 一張圖記住

三個問題，每條工作流都要回答

01 · Input
資料從哪來
表單、CSV、Webhook、排程、人工貼上。每段流程的起點，必須說得出「來源、格式、頻次、必要欄位」。
02 · Process
AI 做什麼
分類 / 摘要 / 改寫 / 抽取。一段流程一個 AI 動作，prompt + model + 期望輸出格式三件套要寫死。
03 · Output
結果放哪
Sheets、Notion、Slack、Email。終點是「會被人或下一段流程接住」的地方，不是控制台。

工作流的核心問題不是「AI 能不能做」，而是你能不能把這件事說清楚。能寫成文件的，就能被標準化。

Common Mistake · 學員最常踩的雷

選錯任務做自動化，越做越累

適合 · 入選

高頻 · 格式固定 · 失敗成本低

  • 每週 / 每月固定要做
  • 輸入是 CSV / 表單 / 結構化欄位
  • AI 跑錯不會立刻爆事故
  • 例：社群留言情緒分類
不適合 · 暫緩

每次都要看情況 · 錯了代價大

  • 合約校稿（錯一字賠錢）
  • 感謝信（一年寫一次）
  • 活動挑圖（高度主觀）
  • 需要看臉色判斷的事

判斷捷徑：能寫出別人能照做的 SOP，就能標準化。每次都要「看情況」，先別自動化。

08 / 38 講義 · CH3-2
The Four · 寫一次 / 用半年

四個要素，三個月後的自己看得懂

01

輸入規格

來源（粉專後台 / 表單 / API）、觸發（每週一 09:00 排程 / Webhook）、欄位定義（timestamp · string · url）、驗證規則（必填欄缺 → 拒收）。

02

處理邏輯

完整 prompt（含角色 / 任務 / 輸出格式）、模型（gpt-4.1）、temperature（0.2）、max_tokens（200）。能複製貼上就跑得起來的程度。

03

輸出格式

目的地（Google Sheet「已分類」分頁）、欄位對應（A=原留言、B=sentiment、C=topic）、命名規則、版本標記。

04

例外處理

空訊息 → 落 other / neu；客訴比例 > 20% → Email 通知主管；token 超量 → 分批重跑。不能整個崩掉。

Failure Modes · 失敗路徑要先想
11 / 38 講義 · CH3-2
Pre-mortem · 工作流上線前必做

AI 出錯時，整條鏈不能跟著崩

情境 1
分到不存在的類
偵測：分類不在白名單；備案：標記 unknown，落人工分頁，每日 Email 給負責人。
情境 2
漏分某客戶
偵測：輸入筆數 ≠ 輸出筆數；備案：缺的批次自動重跑一次，仍缺 → 告警 + 該客戶標 manual。
情境 3
token 中途用完
偵測：API 回 429 / context_length；備案：分批切半 + 從上次斷點續跑，不能整批重來。
原則：每個失敗情境都要有「怎麼自動偵測 + 怎麼回退到上一步」，不能整條流程崩潰重來。

Webhook · 工作流的接線盒

Webhook 不是技術，是一條 URL

概念

Make / n8n 給你一條 專屬 URL，外部系統（表單、付款、CRM）把資料 POST 到這條 URL，你的流程就會醒過來開始跑。

不用 polling、不用排程、不用問「資料來了沒」——來了它會敲你的門。

最小範例 · curl 測試
curl -X POST [YOUR_WEBHOOK_URL] \
  -H "Content-Type: application/json" \
  -d '{
    "submission_id":"FM-20260420-0031",
    "customer":{
      "name":"王小姐",
      "email":"x@example.com"
    },
    "issue":{
      "priority":"high",
      "content":"訂單未到"
    }
  }'
第一步：在 Make / n8n 建好 Webhook 模組 → 複製 URL → 貼到觸發來源（表單通知 / 付款 webhook 設定頁）→ 這條路就通了。

19 / 38 講義 · CH3-3
Acceptance Criteria · 5 項驗收

每接一條 Webhook，跑一次這份清單

01

狀態碼 200

Webhook 顯示「Accepted」，沒回 200 代表外部系統根本沒成功送進來。

02

必填欄位有值

customer.name / issue.priority / submitted_at 不能是 undefined / null / 空字串。

03

枚舉值正確

issue.priority 必須是 high / medium / low 其中之一，不能是「緊急」「很急」「！！！」。

04

時間能解析

submitted_at 是有效 ISO 8601（2026-04-20T09:12:33+08:00），不是「4/20 早上」。

05

secret token 驗證

payload 含 secret 欄位，流程第一步驗 token，不對就丟棄。Webhook URL 一旦洩漏任何人都能 trigger。

Debug · 流程失敗的三種原因
20 / 38 講義 · CH3-3
Debug Heuristics · 學員最常卡的三個地方

跑不動時，先別怪 AI

原因 1
資料對不上
欄位名拼錯、JSON 結構與想像不同、Sheets 第一列沒當欄名。先到 Run History 看 module 1 的輸出長什麼樣。
原因 2
API 回非 JSON
prompt 沒鎖死格式，AI 偶爾回「好的，以下是分類結果...」。加 response_format=json_object 或在 prompt 寫死範本。
原因 3
限流 / 超時
短時間打太多次 OpenAI、batch 太大 token 爆。加 Retry 模組（最多 3 次，exponential backoff）+ 分批。
除錯心態：一次只改一個地方，確認後再改下一個。同時改多處，下次遇到同樣問題還是不會。

SOP in 3 Steps · 三步驟生 SOP
31 / 38 講義 · PRAC3-4
The Method · 用 AI 把口語做法變 SOP

不用先整理，用講的就好

Step 1 · 選任務

最熟、重複最多次

不要選「最重要的」——選「閉著眼睛都會做的」。重複性越高，SOP 價值越大。

Step 2 · 口語給 AI

直接說 / 不要先整理

「我每週五要做 XXX，步驟大概是這樣⋯」口語比填表更完整，AI 會幫你結構化。

Step 3 · 補覆核 + 備案

「請補出人工覆核節點」

初版 SOP 通常只有步驟。加一句讓 AI 補覆核點 + 失敗備案——這是 SOP 真正可用的關鍵。

核心心法：SOP 不是寫給自己看，是讓別人能接手。判斷標準：把這份交給剛進公司的人，他能不能照做？

Pre-mortem · SOP 失敗的兩個地方

初版 SOP 的致命兩個缺欄

缺 1 · 覆核節點

AI 做完直接往下走

最容易出問題的地方：沒人確認 AI 輸出是否正確。把覆核節點寫進 SOP，才是安全可交付的工作流設計。

補法：每個 AI 節點後加一格「誰看 / 看什麼 / 看完按什麼」。

缺 2 · 失敗備案

「正常情況下」就完了

SOP 只寫快樂路徑，異常一發生就沒人知道下一步。每個可能失敗的點都要補一行備案。

補法：對 AI 加一句「請在每個可能失敗的步驟後補上備案路徑」。
