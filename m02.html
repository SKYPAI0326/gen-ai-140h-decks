Version A · 模糊

「幫我寫一封 Email。」

格式 AI 自己決定。
語氣 AI 選最通用的。
長度 AI 寫到它覺得夠。
每次結果都不同，無法複用。

Version B · 精確

「你是有 6 年招募經驗的 HRBP……」

格式直接貼用。
語氣符合情境。
長度不多不少。
同個 Prompt 可以穩定複用。

— Premise —
The Formula · 把要素背下來
06 / 38 講義 · CH1-2
WHO · WHAT · CONSTRAINT · FORMAT

四要素就是點餐。

角色＝廚師類型／任務＝菜名／限制＝過敏原／格式＝裝盤方式—— 你說得越清楚，端出來的菜越接近你要的。

01 · WHO 角色
你是誰？指定角色讓 AI 切換思維框架——律師、設計師、HR、行銷的寫法完全不同，是控制語氣與專業度最簡單的方式。
02 · WHAT 任務
要做什麼？用動詞開頭——「摘要」「改寫」「分類」「比較」。任務越具體，輸出越可預期。
03 · CONSTRAINT 限制
不能做什麼？字數上限、不能提競品、不加客套話——限制是品質的護欄，沒有限制的 AI 很容易過度發揮。
04 · FORMAT 輸出格式
長什麼樣？最常被忽略卻影響最大。表格／JSON／Email／條列——沒指定，AI 就替你選，而它選的往往不是你要的。

Output Formats · 6 種你今天就用得上的
10 / 38 講義 · CH1-2
Format Cheatsheet · 一頁速查

每個格式背後都有對應的下游工具。

選格式，本質是選「這份輸出最後要被貼進哪裡」。

Markdown 表格
適合貼 Notion／Google Sheets／可篩選排序的場景。寫法：「輸出為 Markdown 表格，欄位：A｜B｜C。」
JSON
適合接 n8n／Make／API。寫法：指定 schema，並結尾加「只輸出 JSON，不要其他說明文字」。
SOP 條列
適合操作流程／教學步驟。寫法：「以編號步驟列出，每步驟不超過 2 句說明。」
Email 格式
適合對外溝通。寫法：「主旨行＋稱謂＋正文＋署名，正式語氣。」
條列重點
適合摘要／會議紀錄／Slack。寫法：「最多 5 點，每點動詞開頭，不超過 20 字。」
簡報頁綱
適合提案／課程／報告骨架。寫法：「每頁標題＋3 個要點，共 8 頁，第一頁封面。」

Few-shot · 給例子比給說明更有效
11 / 38 講義 · CH1-2
Few-shot · 用例子取代解釋

當文字描述講不清楚，
就直接給例子。

Zero-shot vs Few-shot
Zero-shot：「請用簡短、有行動感的語氣改寫以下句子。」
Few-shot：「原句：我們計畫推出新產品。 → 新品即將上市，敬請期待。
原句：本次活動歡迎各界人士報名。 → 名額有限，立即搶先報名！
現在請改寫：本季業績表現優異。」

最佳實務：2–5 個例子。太少沒有規律，太多 AI 容易死板複製。 選你最想要的那種結果當例子——AI 模仿的就是它「看到的最好那版」。

Chain-of-Thought · 讓 AI 先想再答
12 / 38 講義 · CH1-2
CoT · 思維鏈基礎

對需要推理的任務，
讓 AI 先攤開它的思考過程。

技巧 01
「請逐步說明你的推理過程」——讓 AI 把判斷邏輯攤開，你也能驗證它的推理是否合理。
技巧 02
「先分析，再給結論」——迫使 AI 不要跳結論，對複雜問題特別有效。
技巧 03
「請先整理你的假設，再回答」——對有歧義的問題，先讓 AI 說清楚它的理解，再給答案。

註：CoT 會讓輸出變長變慢——只用在需要推理的任務。 只是要產一段文字的場合不需要，那只是浪費 token。

Diagnose first · 不是每次都要重寫

當 AI 輸出不對，
幾乎都出在某一個面向。

先學會診斷問題類型，才能精準修—— 不是每次都需要重寫整個 Prompt，往往只要補一行就夠。

太空泛 ／ 太冗長 ／ 格式錯 ／ 不準確 ／ 不照角色
Page 15 · 五種病因導論
— Diagnose, then fix —
5 Bugs · 對症下藥
16 / 38 講義 · CH1-3
Five Bugs Cheatsheet · 一頁全部修法

看到症狀，立刻知道補哪一行。

Bug 01 · 太空泛
症狀：輸出籠統、沒有具體資訊。修：加「請給具體數字、案例或步驟」，或用 few-shot 給範例。
Bug 02 · 太冗長
症狀：輸出過長、摻雜廢話。修：結尾加「不超過 X 字」「只輸出核心，不要前言和總結」。
Bug 03 · 格式錯
症狀：格式不是你要的。修：「以 Markdown 表格輸出」「輸出純 JSON，不含說明文字」。
Bug 04 · 不準確
症狀：內容有錯或與事實不符。修：「不確定就說不知道，不要猜測」「請標注信心程度或來源」。
Bug 05 · 不照角色
症狀：AI 沒有維持你設定的角色或語氣。修：角色描述移到最前面 + 「全程維持角色，不要跳出說明你的想法」。

Iteration Rhythm · 先對 → 再好 → 再快
17 / 38 講義 · CH1-3
Three-pass Iteration · 不要一次到位

不要在第一輪就要求
正確且漂亮且精簡。

Pass 01
先對
確認方向、內容、事實正確。第一輪只追求「方向沒跑掉」，不管漂不漂亮。
Pass 02
再好
改善語氣、格式、細節品質。方向對了，這一輪把它打磨到可以直接用。
Pass 03
再快
把成功的 Prompt 存到 Notion／Google Doc 模板，下次同類任務直接複用。
Page 17 · 三輪修正節奏
— Iteration is the skill —
Sample · 三輪改 Prompt 的真實樣子
18 / 38 講義 · CH1-3
Sample · 寫一封提案開發信

同個任務、同個 AI——修法不同，結果差三倍。

v1 · 第 0 輪草稿
「幫我寫一封給客戶的提案開發信，介紹我們的企業訓練服務。」結果：通用行銷信，無客戶資訊、無痛點連結、無下一步。
v2 · 先對（補方向）
補上角色、客戶背景、痛點（3–6 月養成期）、email 目的（約訪談）、必含三點。改善：方向對了、有打中痛點，但語氣偏死板、結尾制式。
v3 · 再好（補品質）
加字數 250–300 字、口語化每句不超 25 字、開頭不用「您好」、禁用「賦能 / 提升效能 / 解決方案」、結尾兩個時段給對方選。結果：可直接寄出。

v1 → v2 補的是方向，v2 → v3 補的是品質，v3 存檔變模板—— 三輪修的不是同類問題，每輪解決的是不同層級的失敗。

三輪症狀演化
v1：散文式概要（太空泛）

v3：補 5 條 + JSON → 為了湊數編造 team retreat、brand guidelines（不準確 + 格式錯）

v4 Rules · 對症下藥的 6 條規則
20 / 38 講義 · CH1-3
v4 · 把「先對」補回來

每條規則都對應到一個具體病因——
不是「我覺得這樣比較好」。

Rule 01 · 角色定位
「你是會議助理，只整理會議裡明確說過的事，不得編造。」（修：不準確）
Rule 02 · 待辦定義
「待辦 = 動作 + 負責人 + 大概截止，且會議中明確被決議的事項。」（修：太空泛）
Rule 03 · 不得補常識
「禁止憑『公司應該做的事』推測（如 team retreat、brand guidelines）。」（修：不準確）
Rule 04 · 缺資訊處理
「owner / deadline 沒明說 → 填 null；禁 TBD、Q2、Next month 這種模糊時間。」（修：格式錯）
Rule 05 · 語言一致性
「task 用中文，其他欄位英文（符合 Asana 慣例）。」（修：格式錯）
Rule 06 · 未決議分流
「『待議』『討論未結論』放 pending_decisions 陣列，不塞 tasks 湊數。」（修：不準確）

Self-Review · 讓 AI 自己批評自己
21 / 38 講義 · CH1-3
Self-Review · 你不一定要親自重寫

讓 AI 批評它自己，
再寫一個改善版。

Step 01 · 自我檢查
「請檢查你剛才的輸出，有哪些地方不夠準確、不夠清楚、或不符合我的需求？列出 3 個問題。」
Step 02 · 自我重寫
「根據你列出的問題，重新寫一個改善版本。」AI 帶著批評意識重寫，品質通常比第一版高。
Step 03 · 版本比較
「請給我版本 A 和版本 B，並說明你認為哪個更好及原因。」讓 AI 給你多個選擇，由你做最終決定。

Stability · 同個 Prompt 跑 3 次

結果差很多 =
Prompt 還需要更多限制。

把同一個 Prompt 連跑 3 次，比較穩定度—— 如果三次結果差很大，不是 AI 在變動，是你的限制條件還不夠多。

關鍵心態
在 AI 時代，「修改提示詞」本身就是一種工作技能。
第一次沒成功不是失敗——那只是第一輪草稿。

How to Fill · 範本素材：催款信
26 / 38 講義 · PRAC1-1
Worked Example · 把模糊任務逐欄填好

場景：寫一封催款信給合作客戶。

角色 Role
你是有 8 年經驗的設計工作室行政窗口，熟悉 B2B 帳款往來禮節，正式但帶人情味，不打官腔也不卑微。
任務 Task
撰寫催款郵件，對象是合作客戶「晨光文創」的行銷專員王小姐，合約金額 45,000 元，交付一個月尾款未到。
限制 Constraint
180–250 字；不用「催繳／逾期」強硬字眼；顧到長期合作語感；結尾留台階；提供兩個具體下一步（轉財務／回信確認日）。
輸出 Output
Email 格式：主旨（15 字內）＋ 正文分三段 ＋ 結尾署名「弄一下工作室 行政部」。不加額外前言。

四欄填好 → 點組合 → AI 直接產出可寄出的信。 換場景時只改「任務 + 限制」具體內容，骨架不動。

Templates · 不知道從哪開始？點範本

四個內建範本，
覆蓋 80% 的上班族任務。

5 Dimensions · 五維度健診
31 / 38 講義 · PRAC1-2
Health Check · 貼入 Prompt 自動診斷

每個 Prompt 都有五個體檢項目。

Dim 01 · 角色 Role
有沒有指定職業 / 產業 / 語氣風格？只寫「你是專家」會被扣分。
Dim 02 · 任務 Task
動詞是否明確？平台 / 目的 / 受眾是否說清楚？
Dim 03 · 背景 Context
給誰看 / 什麼情境用 / 上下文資訊是否充分？
Dim 04 · 限制 Constraint
字數、禁用詞、必含元素（CTA / hashtag）是否規範？
Dim 05 · 輸出 Output
輸出能否直接貼進下游工具，不需要人工再編排？

4 Failure Modes · 四種典型失分案例
32 / 38 講義 · PRAC1-2
Failure Modes · 四種一定會犯的錯

每種錯都對應一個維度的低分，且都有具體修法。

Case A · 缺角色
「請幫我寫一段關於遠距工作優缺點的文章……」結果：角色 15 分。修：補一句「你是熟悉混合辦公的 HR 顧問」直接從通用八股逼成顧問視角。
Case B · 缺限制
「你是社群編輯，幫我寫 IG 貼文，主題是深夜食堂……」結果：限制 25 分。修：加字數 / 鉤子 / CTA / hashtag 規格 → 直接拉到 90。
Case C · 缺脈絡
「幫我寫一封信。」結果：總分 25 分（重寫等級）。修：補齊「寫給誰／為什麼寫／你的身份／希望對方做什麼」五維度同時跳 30–50 分。
Case D · 過度複雜
一次堆 10+ 條要求。陷阱：健診器 70 分，但實際輸出 30 分（AI 隨機挑 3-5 條執行）。修：拆成 3-4 條獨立 Prompt 分次跑。
