05 / 38 講義 · CH2-1
The Four · 一張圖記住

四個動作，知識工作的基本動作

01 · Summarize
摘要
把長變短。會議錄音、報告、文件、收件匣——壓出三句話。
02 · Classify
分類
把雜變齊。客戶意見、待辦、檔案、訊息——丟進對的桶子。
03 · Rewrite
改寫
把硬變軟，或把軟變硬。同一份內容改給不同受眾。
04 · Expand
擴寫
把骨架變成成品。大綱、要點、關鍵字——展開成完整段落。

這四個動作能組合出 80% 的日常文字工作。剩下 20% 是這四個的多輪疊加。

Summarize vs Rewrite · 最常被混淆的兩個
06 / 38 講義 · CH2-1
Common Confusion · 學員最常混淆

摘要不是改寫，動的東西完全不同

Summarize · 摘要

資訊量變少 · 受眾不變

  • 抽掉次要、保留核心
  • 長度與顆粒度可控
  • 字數常壓到原文 10-30%
  • 給「同一個人」的精簡版
Rewrite · 改寫

資訊量不變 · 受眾換人

  • 內容守住、語氣換掉
  • 對主管 / 客戶 / 同事 / 新人
  • 字數可長可短
  • 給「不同的人」的同一份內容

搞混的代價：學員常用「改寫」 prompt 期望「摘要」結果，得到的是等長度的同義改寫，白做兩次。

Summarize · 三個精度控制器
07 / 38 講義 · CH2-1
Pipeline · 摘要交辦單必填三欄

三個控制器，決定摘要長相

不講這三個，AI 會給你一份「長度感人但抓不到重點」的安全牌摘要。

Three Knobs · 摘要的三個精度旋鈕
01 · Length
字數 / 句數
「壓到 3 句話」「200 字以內」。沒寫長度，AI 會自己取一個保險的中間值。
02 · Grain
顆粒度
「只留結論」「結論 + 三個論點」「事件時間軸」——告訴它你要看哪一層。
03 · Frame
視角 / 角色
「給沒參加會議的主管看」「以技術主管視角」——同份原文摘出來會完全不同。
實作習慣：三欄少一欄，回去補。三欄都填，第一版可用率超過 80%。

08 / 38 講義 · CH2-1
Four Audiences · 一份內容，四種講法

同一份白皮書，四個版本

A · 主管
電梯三句話
結論先、影響先、要錢要人放最後。動詞要硬：「決定」「投入」「砍掉」。
B · 客戶
利益翻譯
把「我們做了什麼」改成「您會得到什麼」。技術名詞拆成日常語言。
C · 同事
分工備忘
誰要動 / 動什麼 / 何時交。可以保留術語，省鋪陳。
D · 新人
入門導讀
補背景、補名詞、補「為什麼這樣做」。可加類比與比喻。

練習方法：拿你上週寫過的一段話，請 AI 改成這四個版本——看哪個受眾最不像你平常的寫法，就是你最該補的能力。

09 / 38 講義 · CH2-1
Pipeline · 擴寫五步

擴寫不是把字變多，是把骨架長肉

初學者常把擴寫當「請寫長一點」，結果是注水文。

Skeleton → Draft · 五步
01 · Skeleton
給骨架
大綱要點 + 段落順序。不交出骨架，AI 自己找順序。
02 · Frame
指定文體
公告 / 報告 / 文章 / 提案。文體決定段長與語氣。
03 · Audience
指定受眾
給誰看，決定要不要鋪背景、要不要解名詞。
04 · Scan
先掃主旨
第一版出來，先看每段第一句對不對，不要先看文字品質。
05 · Polish
局部微調
主旨對了，再請 AI 修語氣、修例子、修數字。一次只改一段。
實作技巧：「掃描主旨 + 局部微調」比「整段重寫」效率高得多。抓 1-2 輪微調，不要期待第一版完美。

10 / 38 講義 · CH2-1
Two Rules · 分類設計

分類比想像中挑桶子的設計

Rule 01 · 互斥

桶子之間不重疊

客戶意見「對價格不滿」與「對折扣不滿」會搶同一筆——AI 會兩邊都丟，下游統計就翻倍。

設計時自問：同一筆能不能只進一桶？

Rule 02 · 窮盡

必有一桶能放

沒設「其他」桶，AI 會硬塞進最像的桶子，造成假分類——下游會以為某類爆量。

設計時自問：奇怪案例會被丟去哪？

Watch Out · AI 文字產出三個盲點
13 / 38 講義 · CH2-1
Three Blind Spots · 三個一定要校對的位置

三個地方，AI 不會自己幫你檢查

Blind 01
數字 / 日期
摘要會自己「圓化」數字（87% 變 90%）、改日期。數字一律回原文比對。
Blind 02
人名 / 角色
A 講的話會被掛到 B 名下。會議紀錄與訪談摘要的高頻錯誤。
Blind 03
否定 / 條件
「不」「除非」「如果」常被吃掉，導致語意倒過來。改寫時最常出。
習慣：把這三項做成自己的 checklist——任何要對外送出的 AI 文字，先掃過這三點再按送出。

Email · 四種類型模板化
16 / 38 講義 · CH2-2
Four Email Types · 你 90% 的信都是這四種

四種 Email，各有專屬骨架

A · 通知 / Notice
事件 + 影響
主旨即結論。一句話講「發生什麼」，一句講「對你影響」，一句講「要做什麼」。
B · 請求 / Request
背景 + 具體要求
先給對方理解動機的最少資訊，再講「我要什麼 / 何時要」。
C · 回覆 / Reply
立場 + 理由
先講「同意 / 不同意 / 部分同意」，再給理由。不要先鋪墊一大段。
D · 婉拒 / Decline
感謝 + 邊界
先謝對方提出，再講邊界（為何不行 / 何時可能行）。不要假設對方會懂言外之意。

這四個骨架做成 4 段 prompt 模板，存進你的模板庫——以後每次寫信，只填入背景就出第一版。

Email · 模板化的五步

Pipeline · 從一封信到一個模板

從每次重寫到填空就出

模板化不是少寫字，是把判斷力沉澱下來，下次照樣管用。

Email Templating · 五步
01 · 抓樣本
挑 3 封信
同類型的 3 封實際信件——你寫過或想學的。AI 從樣本反推共同骨架。
02 · 抽骨架
請 AI 抽結構
「這 3 封信的共同段落順序是什麼？哪些可填空、哪些固定？」
03 · 標變數
變數標明
把「對方姓名 / 事件 / 日期 / 行動」標成 [變數]。下次填空就出。
04 · 入庫
存進模板庫
命名為「通知-001」「請求-002」。Notion / Drive / 文字檔都行，重點是回得來找得到。
05 · 試跑
下次套用
下封同類型信，先套模板出第一版，再微調。第二次起就是省時間。

Meeting Notes · 從逐字稿到可用版本
18 / 38 講義 · CH2-2
Notes Conversion · 兩種版本對齊兩種讀者

會議紀錄不是一份文件，是兩份

A · 紀錄版

給沒參加的人看

  • 議題 → 結論 → 理由
  • 不必寫誰說了什麼
  • 用客觀第三人稱
  • 長度可以長
B · 待辦版

給要動手的人看

  • 誰 / 做什麼 / 何時前
  • 可勾選的動作清單
  • 沒指派的事項要標出來
  • 越短越好

多數會議紀錄之所以沒人看，是因為這兩份混在一起——讀紀錄的人找不到結論，要動手的人找不到自己的事。

Meeting · 從錄音到雙版本的 5 步
19 / 38 講義 · CH2-2
Pipeline · 會議產線五步

五個動作，會議結束 30 分鐘內出兩份

Audio → Notes + Tasks · 五步
01 · Transcribe
轉逐字稿
內建錄音 / Otter / Whisper。重點是有講話人標記。
02 · Cluster
分議題
先請 AI 把逐字稿切成議題塊。每塊一個小標。
03 · Notes
出紀錄版
每塊議題寫「結論 + 理由」，用第三人稱。
04 · Tasks
出待辦版
掃逐字稿找「我會 / 我來 / 下週交」這類動詞，組成誰 / 做什麼 / 何時前三欄表。
05 · Flag
標未指派
沒明確指派的事項要標「待認領」——這是會議紀錄最常掉的東西。

Weekly · 週報快速生成框架
20 / 38 講義 · CH2-2
Weekly Report · 三段就夠

週報三段，30 分鐘出稿

Section 01
本週成果
完成的事 + 數字結果。AI 從你的待辦清單 + 行事曆 + commit 自動萃取。
Section 02
阻礙與決策
卡在哪 / 等誰 / 需要什麼決策。寫給主管看，不要假裝沒事。
Section 03
下週重點
不超過 3 件。寫了第 4 件就回頭砍。重點不是清單長度，是承諾密度。

把這三段變成 prompt 模板，每週只要丟「這週完成的事」當輸入，第一版 5 分鐘就出。

Prompt Sample · 週報模板

Sample Prompt · 直接抄

週報模板，抄回去就能用

// Prompt
你是我的週報助理。我會丟給你「本週發生的事」清單，
請依以下三段格式輸出，每段不超過 4 行：

1. 本週成果：完成什麼、影響什麼。每點一句話，附數字。
2. 阻礙與決策：卡在哪、需要誰決策。沒卡寫「無」。
3. 下週重點：最多 3 件，每件一句話加交付日期。

語氣務實、不浮誇。不確定的事用「待確認」標出。
[這裡貼本週發生的事]
升級：把這段 prompt 存到自己的模板庫，下週換掉最後一行就能再用。再進階就把它寫進 System Prompt，連 prompt 都不用每次貼。

From Person to Org · 從個人省時間到組織一致

文風一致化是組織的隱性資產

Without · 沒做

每個人各自為政

  • 對外文件像不同家公司寄的
  • 新人要學三個月才像「自己人」
  • 客戶看了會問「你們有沒有溝通過」
  • 每位資深員工離職都帶走一套標準
With · 做了

三五條規則進 System Prompt

  • 對外稱呼 / 是否用敬語 / 數字格式統一
  • 避用詞 + 替代詞清單
  • 所有草稿丟 AI 校準一輪再送出
  • 新人從第一週就「像自己人」

進階應用：把組織語言標準整理成 3-5 條規則進 System Prompt。從「個人省時間」升級到「提升組織文字品質」。

內部公告 + 對外文宣兩版。內部硬資訊、對外加情緒鉤子。
Template 03
回覆抱怨

婉拒提案
感謝 + 邊界 + 替代路徑。讓對方下次知道怎麼提才會被接受。

每個模板要交：1 段 Prompt + 1 個填空表 + 1 個示範輸出。三件齊備才算「進得了模板庫」。

Library · 四層結構
L1 · Prompt
Prompt 本身
含角色、任務、輸出格式、約束。能直接複製貼上。

填空表

Self-Drill · 自我演練

把同一份內容
盲改四個版本

拿你上週寫過的一段話，請 AI 改成「對主管 / 對客戶 / 對同事 / 對新人」四個版本。

然後把四個版本印出來，請同學或自己過 24 小時後，盲猜「哪個是給誰的」。

判讀：猜不出來，代表那個 prompt 沒抓到受眾差異。這是免費的反饋訊號。

Four Meeting Types · 四種典型會議
32 / 38 講義 · PRAC2-2
Four Patterns · 四種會議的拆法不同

不同會議，抓的東西不一樣

A · 例會
進度 + 阻礙
每人「上週做了 / 卡哪 / 下週做」三欄。固定欄位、低變化。
B · 決策會
議題 + 結論
每議題：選項 → 決策 → 理由 → 後續行動。沒結論的議題要明確標「未決」。
C · 腦力激盪
想法 + 收斂
先全部點子分群，再標「下一步驗證 / 暫存 / 不採」三類。
D · 一對一
承諾 + 追蹤
雙向承諾要分開列：你答應對方的、對方答應你的。下次會議追蹤。

每種會議要做不同的 prompt。把它做成 4 段，下次會議結束直接配對。

Pipeline · 從紀錄到清單

五步拆出可勾選的待辦

Notes → Actionable Tasks · 五步
01 · Mark
標出動詞
「我會」「我來」「下週前」這些是任務候選。先全部框起來。
02 · Owner
補負責人
每筆任務必有負責人。沒寫的標「待認領」，不要默認是會議召集人。
03 · Deadline
補時間
沒明說的補「下次會議前」。寫「盡快」等於沒寫。
04 · Verb
改成可勾選
「研究 X」改成「整理 X 的三個方案，附比較表」——必須有可驗收的產出。
05 · Next
產出下次議程
所有「待認領」「未決」直接成為下次會議第一段議程，不用重新討論。

// Prompt
你是會議紀錄助理。我會丟一份會議逐字稿，請輸出兩段：

【紀錄版】給沒參加的人看。每議題寫：議題 / 結論 / 理由。

· 沒指派的標「待認領」，不要默認召集人。

[這裡貼會議逐字稿]
